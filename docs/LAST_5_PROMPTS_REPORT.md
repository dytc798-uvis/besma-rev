# 최근 5개 프롬프트 작업 상세 보고서

작성일: 2026-03-20  
대상: BESMA 로컬 MVP

---

## 1) DeprecationWarning 최소 수정 대응

### 요청 요지
- 전체를 크게 건드리지 않고 경고(특히 `datetime.utcnow()`)를 최소 범위로 처리.

### 수행 내용
- 공통 유틸 `utc_now()`를 도입해 `datetime.utcnow()` 대체.
- 핵심 사용처를 점진적으로 교체.
- 테스트 경고 필터를 `pytest` 실행 방식에 맞게 조정.

### 결과
- 런타임/테스트 환경에서 관련 경고가 실사용 기준으로 크게 감소.
- 기존 구조를 유지하면서 최소 수정 원칙 준수.

## 2) 운영형 문서 취합/모니터링 대시보드 골격 구현

### 요청 요지
- 현장: 기간별 제출 대상 문서를 보고 업로드 가능해야 함.
- 본사: 현장별 제출률/미제출/승인대기/반려를 한눈에 보고 승인/반려 가능해야 함.

### 수행 내용
- 백엔드:
  - 문서 요구사항 상태 조회, HQ 대시보드, 히스토리, 배지 API 추가/확장.
  - 반려 후 재업로드 이력(`DocumentUploadHistory`) 추적 로직 반영.
  - 주기/기간(day/week/month/quarter/half_year/year/event/all) 처리 보강.
- 프론트:
  - 현장 문서 대시보드 페이지 구현(탭, 상태 요약, 업로드/재업로드/이력).
  - HQ 문서 대시보드 페이지 구현(현장 중심 좌우 패널, 검토/승인/반려).
  - 메뉴 배지 연동.

### 결과
- “생성형 문서” 수준에서 “운영형 취합/모니터링”으로 승격되는 핵심 흐름 확보.
- 반려→재업로드→이력 확인까지 시연 가능한 골격 확보.

## 3) 로그인 후 persona-select 재진입 방지(운영 동선 분리)

### 요청 요지
- 로그인 성공 후 운영 계정이 다시 `persona-select`로 가지 않도록 최소 범위 수정.
- dev/test 전용 동선은 유지 가능하되 운영 흐름과 분리.

### 수행 내용
- 라우터 가드와 로그인 후 리다이렉트 분기를 정리.
- `persona-select`를 dev 전용 경로 의미로 제한.
- 운영 계정은 역할/권한 기반 목적 화면으로 바로 이동하도록 조정.

### 결과
- 운영 로그인 UX에서 불필요한 재진입 제거.
- 테스트 전환 동선은 유지하면서 운영 동선과 충돌하지 않도록 분리.

---

## 4) SQLite 기준 DB 식별 및 “70개 현장 미노출” 원인 판정

### 요청 요지
- 현재 실행 중인 DB를 확인하고, 70개 현장 데이터가 왜 안 보이는지 코드/DB 근거로 판정.
- 확인 전 무리한 수정 금지.

### 수행 내용
- API/DB 상태와 시드/데이터 적재 상태를 대조 검증.
- 화면 필터/코드 문제인지, 데이터 부재인지 근거 기반으로 분리 분석.

### 결과(판정)
- 문제 핵심은 “표시 로직 오류”보다 “활성 DB에 70개 데이터가 실제 적재되지 않은 상태”로 판정.
- 대규모 코드 변경 없이 원인 중심으로 보고.

---

## 5) HQ에는 반려 표시되는데 Site에는 대상 없음 정합성 복구

### 요청 요지
- 반려 1건이 HQ→Site(재업로드 대상)로 이어지도록 우선 복구.
- 전체 화면 대수술 없이 최소 수정.

### 수행 내용
- 백엔드 상태 계산 로직 보강:
  - 현재 기간에 문서가 없더라도, 최근 반려 건을 재업로드 대상 후보로 노출하도록 보완.
- 관련 통합 테스트 시나리오 보강(이전 날짜 반려 건의 현재 날짜 노출 확인).
- 환경 이슈(구버전 서버 기동)도 분리 진단.

### 결과
- 코드상 정합성 복구 및 테스트 통과 확인.
- 단, 실제 HTTP에서 반영되려면 백엔드 재기동이 필요하다는 운영 조건 명확화.

## 종합 결론

- 최근 5개 프롬프트는 공통적으로 **최소 수정/운영 시연 가능성/근거 기반 진단** 원칙으로 수행됨.
- 현재 상태는 다음을 충족:
  - 운영 로그인 동선 정리
  - 문서 운영 대시보드 핵심 골격 확보
  - 반려-재업로드 흐름 정합성 복구
  - 데이터 미노출 이슈의 원인 판정 완료
- 잔여 리스크는 코드보다는 실행 환경(서버 재기동, 실제 DB 적재 상태) 측면이 큼.
