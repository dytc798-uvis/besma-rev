# 근로자의견청취 / 부적합사항 — 현장·본사 운영 설명서

> 공지·교육자료로 복사해 사용 가능합니다.  
> **시스템 동작**은 [DECISION-060] 및 `risk_gates.py`의 `ready_for_risk_db` 조건과 일치하도록 작성했습니다.

---

## 시스템과 문서의 차이 (한 줄)

- **시스템이 보장하는 것**: 접수 ≠ DB 승격, DB 승격은 **현장「DB 등록 요청」+ 본사「DB 등록 승인」**이 모두 있어야 `ready_for_risk_db`가 됩니다.
- **문서/운영에서 권장하는 것**: 아래 「요청할 때 / 말 때」는 **판단 가이드**이며, 조치 완료 여부는 **HQ 승인 API에서 강제 검증하지 않습니다** (운영 교육용 기준으로 두면 됩니다).

# 1. 현장용 1장 설명서

## 근로자의견청취 / 부적합사항 처리 방법 (현장)

### 1. 기본 흐름

```text
의견(또는 부적합) 등록 → 접수/반려 → 조치 상태 관리 → (필요 시) 위험성평가 DB 등록 요청
```

- **등록**: 전용 관리대장 화면에서 **파일 업로드**가 기본입니다. (`내 현장 문서`에서는 해당 두 문서 유형 **업로드 불가** — 중복 방지)

### 2. 각 단계 설명

#### ① 접수 / 반려

- **접수**: 현장에서 조치·관리할 가치가 있다고 판단할 때
- **반려**: 부적절하거나 현장 운영상 처리 대상이 아닐 때

👉 기준 예시: “현장에서 실제로 조치·기록할 가치가 있는가”

#### ② 조치 상태

- **미조치** / **조치중** / **조치완료**
- 실제 작업 진행만 반영합니다.

#### ③ (중요) 위험성평가 DB 등록 요청

👉 **접수·조치와 별개**입니다.

**요청을 검토할 때(운영 가이드):**

- 반복 발생 가능성
- 위험성이 큰 작업·프로세스
- 다른 현장에도 적용·공유할 가치

**요청하지 않아도 되는 경우(예시):**

- 단순 민원성
- 일회성으로 현장에서 종료
- 현장 단위로 충분한 사항

### 3. 절대 규칙 (시스템과 동일)

- **접수 ≠ DB 등록**
- **조치완료 ≠ DB 등록**
- **DB 등록(승격)** 은 **별도 버튼·별도 단계**입니다.

### 4. 예시

**✔ DB 등록 요청을 검토할 만한 사례**

- 고소작업 난간 미설치가 반복
- 감전 위험이 있는 작업 순서·장비 문제

**❌ DB 등록 요청이 불필요할 수 있는 사례**

- 휴게실 의자 부족
- 식수 위치 불편

# 2. 본사용 운영 기준

## 위험성평가 DB 등록 승인 (HQ)

### 1. 기본 흐름

```text
현장 「위험성평가 DB 등록 요청」 → 본사 검토 → DB 등록 승인 / DB 등록 반려
```

### 2. 승인 시 시스템이 확인하는 것 (자동)

다음이 **모두** 맞을 때만 본사 **DB 등록 승인**이 가능합니다.

- 현장 **접수** 완료 (`receipt` = accepted, 현장 반려 아님)
- 현장 **DB 등록 요청** 상태 (`requested`)

👉 **현장 요청이 없으면 승인 API는 409로 거절**됩니다.  
👉 **현장 반려 건은 승인 불가**입니다.

### 3. 운영에서 추가로 볼 것 (가이드 — 시스템 미강제)

아래는 **교육·심사용 권장 기준**으로 두고, 필요 시 본사 내부 기준서로 세분화하면 됩니다.

- 조치가 어느 정도 진행되었는지(진행/완료)
- 반복 위험·타 현장 적용 가능성
- 단순 민원·일회성·현장 종료 여부

### 4. “DB 반영”에 대해 (정확한 표현)

- 본사가 **「DB 등록 승인」** 하면, 해당 row는 시스템 상 **위험성평가 DB 반영 후보(`ready_for_risk_db`)** 로 확정됩니다.
- **risk_library 등 실제 라이브러리 테이블로의 자동 머지/반영**은 별도 배치·연동 설계에 따릅니다. 문구로 “승인 즉시 DB 테이블에 자동 삽입”이라고 단정하지 않는 것이 안전합니다.

### 5. 포상 후보 등록

- **DB 등록 승인** 이후에만 포상 후보 등록이 가능합니다. (시스템 제약)
- 우수 개선·확산 가치 등은 본사 운영 기준으로 판단합니다.

# 3. UI 문구 (앱 반영 권장)

## 현장

- **접수** / **반려** / **조치 상태** — 유지
- 버튼 권장명: **「위험성평가 DB 등록 요청」** (`DB 등록`만 단독 사용은 지양)

## 본사

- **「DB 등록 승인」** / **「DB 등록 반려」** (모호한 「승인」「확인」 단독 표기 지양)

## 안내 문구 예시 (툴팁·도움말)

**현장**

> 위험성평가 DB 등록 요청은 반복 위험이 있거나 타 현장 적용이 필요한 경우에만 누르세요. 접수·조치와는 별개입니다.

**본사**

> 현장에서 「위험성평가 DB 등록 요청」을 한 항목만 DB 등록 승인할 수 있습니다.

# 4. 최종 요약

```text
1. 현장: 처리(접수·반려·조치) + 필요 시 「위험성평가 DB 등록 요청」
2. 본사: 「DB 등록 승인」·「DB 등록 반려」·포상 후보
3. DB 반영 후보 확정: 현장 요청 + 본사 승인 (시스템)
   실제 라이브러리 반영: 연동·운영 정책에 따름
```
