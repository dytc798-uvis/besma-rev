# BESMA AI 협업 통합운영규정

## 1. 총칙

### 1.1 목적
본 규정은 BESMA 프로젝트에서 Cursor, Codex 및 기타 AI 코딩 에이전트가 공통으로 따라야 할 운영 기준을 정함으로써 다음 목적을 달성하는 데 있다.

1. AI 작업자 간 작업 기준 통일
2. 세션 누락 및 컨텍스트 혼선 방지
3. 프로젝트 경로 및 프레임워크 오인 방지
4. 코드 수정 범위와 영향 범위 통제
5. 보고 형식과 검증 절차 통일
6. 오류 대응 절차의 표준화
7. 향후 세부 지침 문서의 상위 기준 확립

### 1.2 적용 대상
본 규정은 BESMA 프로젝트에 참여하는 모든 AI 작업자와 이를 활용하는 협업 세션에 적용한다.

### 1.3 적용 범위
본 규정은 다음 업무 전반에 적용한다.

1. 코드 조회, 수정, 생성, 삭제
2. 설정 파일 변경
3. 문서 작성 및 갱신
4. 실행, 테스트, 검증, 오류 분석
5. 작업 보고 및 후속 제안

### 1.4 용어 정의
본 규정에서 사용하는 용어의 정의는 다음과 같다.

1. `AI 작업자`
Cursor, Codex 및 기타 AI 기반 코딩 보조 도구 또는 에이전트를 말한다.

2. `세션`
AI 작업자가 특정 작업 목적을 가지고 사용자와 상호작용하며 작업을 수행하는 단위 시간을 말한다.

3. `워크스페이스`
현재 AI 작업자가 실제로 접근하고 수정할 수 있는 작업 디렉터리 또는 프로젝트 루트를 말한다.

4. `프로젝트 기준경로`
사용자가 작업 대상으로 지정한 프로젝트의 절대경로를 말한다. 경로가 명확히 확인되지 않으면 작업 대상이 확정되지 않은 것으로 본다.

5. `작업범위`
사용자가 이번 세션에서 수정 또는 검토를 허용한 파일, 디렉터리, 기능, 설정의 범위를 말한다.

6. `영향범위`
수행한 변경이 기능, API, DB, 설정, 실행 방법, 문서, 배포, 사용자 흐름에 미치는 범위를 말한다.

7. `기준문서`
프로젝트의 운영, 설계, 구현, 검증, 보고 기준으로 참조하는 문서를 말하며, 본 통합운영규정이 최상위 기준문서가 된다.

8. `외부 조정자(ChatGPT)`
프로젝트 폴더 내부를 직접 수정하지 않고, Cursor/Codex 세션의 맥락을 이해하고 이어가기 위한 상위 조정자 역할을 수행하는 AI를 말한다. 외부 조정자는 설계 기준을 잡고 세션이 끊겼을 때도 다시 복구할 수 있도록 정보를 수집·정리·해석하는 데 중점을 둔다.

9. `인계`
하나의 세션에서 다른 AI 또는 외부 조정자(ChatGPT)에게 작업 맥락을 넘겨주는 행위를 말하며, 최소한의 공통 정보를 포함해야 한다.

## 2. 기본 원칙

### 2.1 추정 금지 원칙
파일 구조, 경로, 프레임워크, 포트, 설정, 실행 방식, API 경로, DB 상태는 추정하지 않는다. 확인되지 않은 사항은 사실처럼 단정하지 않는다.

### 2.2 경로 확인 우선 원칙
모든 작업은 현재 워크스페이스 절대경로와 프로젝트 기준경로를 먼저 확인한 후 수행한다.

### 2.3 프레임워크 확인 우선 원칙
프로젝트가 FastAPI, Vue, Next.js, React 등 어떤 스택으로 구성되어 있는지는 실제 파일 구조와 설정을 통해 먼저 확인한다.

### 2.4 기존 시스템 보호 원칙
현재 동작 중인 기능, 사용자 코드, 기존 설정, 배포 흐름을 불필요하게 훼손하지 않는다. 작업 목적과 직접 관련 없는 변경은 하지 않는다.

### 2.5 실행 가능 상태 유지 원칙
모든 변경은 가능한 한 프로젝트가 계속 실행 가능한 상태를 유지하도록 작은 단위로 수행한다.

### 2.6 작은 단위 변경 원칙
수정은 원인과 목적이 분명한 최소 단위로 수행하며, 여러 문제를 한 번에 광범위하게 뒤섞어 변경하지 않는다.

### 2.7 보고 의무 원칙
모든 작업은 수행 내용, 수정 파일, 영향 범위, 테스트 결과, 미완료 사항을 명확히 보고해야 한다.

### 2.8 검증 우선 원칙
수정 후에는 가능한 범위 내에서 실행, 호출, 테스트, 로그 확인 등 실질 검증을 우선 수행한다.

### 2.9 Invariant Check 의무 원칙
Invariant Check는 선택 절차가 아니라 모든 AI 작업에 강제되는 상시 검수 절차로 운영한다. Invariant Keeper는 별도 역할이 아니라, 모든 AI 작업자가 반드시 거쳐야 하는 공통 검수 기준으로 정의한다.

### 2.10 Instruction Validation Gate 의무 원칙
Instruction Validation Gate는 모든 구현 및 설계 지시 수행 전에 반드시 거쳐야 하는 사전 검증 절차로 운영한다. 이 절차는 Invariant Check 이전 단계의 강제 검증 레이어이며, 잘못된 지시가 정책 위반 코드로 고정되는 것을 방지하기 위한 상시 규정이다.

## 3. 프로젝트 식별 및 경로 확인 규칙

## 3. 프로젝트 식별 및 경로 확인 규칙

### 3.1 프로젝트 구분 원칙
기존 프로젝트와 신규 로컬 MVP는 별도 프로젝트로 간주한다. 이름이 유사하거나 동일한 서비스 명칭을 사용하더라도 동일 프로젝트로 추정하지 않는다.

### 3.2 BESMA 기준 사례
다음 사례는 실제 BESMA 작업에서 반복될 수 있는 대표 예시로 간주한다.

1. `C:\BESMA` = 기존 프로젝트
2. `D:\besma-rev` 또는 `C:\besma-rev` = 신규 로컬 MVP

### 3.3 작업 시작 전 필수 확인
모든 AI 작업자는 작업 시작 전에 아래 항목을 반드시 확인한다.

1. 현재 워크스페이스 절대경로
2. 프로젝트 루트 구조
3. 사용 프레임워크 및 런타임

### 3.4 불명확 시 조치
경로, 프로젝트, 프레임워크 중 하나라도 불명확한 경우에는 코드 수정, 파일 생성, 설정 변경을 진행하지 않는다. 먼저 확인 결과와 불명확한 지점을 사용자에게 보고한다.

### 3.5 오인 사례 반영 의무
과거 세션에서 `C:\BESMA`를 `D:\besma-rev`로 오인하거나, 반대로 신규 MVP를 기존 프로젝트로 오인한 사례가 있었던 경우, 이후 세션에서는 경로 확인을 첫 절차로 간주한다.

## 4. 세션 시작 표준 절차

### 4.1 세션 시작 체크리스트
매 세션 시작 시 AI 작업자는 아래 항목을 확인하고 필요 시 사용자에게 명시적으로 정리하여 공유한다.

1. 현재 작업 대상 프로젝트
2. 현재 워크스페이스 절대경로
3. 프레임워크
4. 실행 포트
5. 오늘 작업 목적
6. 수정 허용 범위
7. 수정 금지 범위
8. 참조 문서 존재 여부

### 4.2 확인 방법
체크리스트 항목은 다음 기준으로 확인한다.

1. 프로젝트와 경로는 실제 터미널 경로와 루트 디렉터리 목록으로 확인한다.
2. 프레임워크는 `README`, `package.json`, `requirements.txt`, `pyproject.toml`, `main.py`, `vite.config`, `src/main.ts` 등 실제 파일로 확인한다.
3. 실행 포트는 환경변수, 설정 파일, README, 서버 실행 명령 또는 코드 내 설정으로 확인한다.
4. 수정 허용 범위와 금지 범위는 사용자 요청과 현재 작업 목적에 근거해 명확히 구분한다.

### 4.3 체크리스트 누락 시 원칙
체크리스트를 완료하지 못한 상태에서는 실제 변경 작업에 착수하지 않는다.

### 4.4 사전 검증 체크
세션 시작 후 실제 구현 또는 설계 변경에 착수하기 전, Instruction Validation Gate 기준에 따라 사용자 지시의 충돌 여부와 해석 가능성을 먼저 점검한다.

## 5. 작업 수행 규칙

### 5.1 작업 전 확인사항
작업 전에는 다음을 확인한다.

1. 수정 대상 파일의 현재 상태
2. 관련 설정, API, 의존 파일 존재 여부
3. 작업 목적과 직접 연결된 최소 수정 범위
4. 실행 또는 테스트 가능 여부

### 5.2 작업 중 준수사항
작업 중에는 다음을 준수한다.

1. 사용자 요청 범위를 벗어나는 구조 변경을 하지 않는다.
2. 변경 이유가 명확한 파일만 수정한다.
3. 설정값 변경 시 실행 방식, 환경변수, 포트, 인증 방식에 미치는 영향을 함께 고려한다.
4. 확인되지 않은 문제를 임의의 가정으로 해결했다고 단정하지 않는다.

### 5.3 작업 후 정리사항
작업 후에는 다음을 수행한다.

1. 변경 파일 목록 정리
2. 변경 내용 요약
3. 영향 범위 정리
4. 테스트 방법 및 결과 정리
5. 남은 위험 또는 미완료 항목 정리

### 5.4 실제 변경 수행 원칙
사용자가 수정 또는 해결을 요청한 경우, AI 작업자는 설명만 제시하고 멈추지 않는다. 특별한 제약이 없는 한 실제 변경, 검증, 정리까지 수행한다.

### 5.5 placeholder 허용 기준
placeholder, mock, TODO, 임시값은 아래 조건을 모두 만족할 때에만 허용한다.

1. 사용자 요청 범위 내에서 임시 조치가 명시적으로 필요할 것
2. 실제 구현 불가 사유가 명확할 것
3. 임시 조치임을 코드 또는 보고에 분명히 표시할 것
4. 후속 근본조치가 무엇인지 보고할 것

### 5.6 임시조치와 근본조치 구분
모든 해결책은 `임시조치`와 `근본조치`를 구분하여 보고한다. 임시조치만 수행한 경우, 왜 근본조치를 즉시 수행하지 못했는지 명확히 남긴다.

### 5.7 과도한 구조 변경 금지
한 번의 세션에서 대규모 리팩터링, 구조 재편, 계층 이동, 다수 모듈 재설계를 동시에 수행하지 않는다. 필요한 경우 별도 작업으로 분리한다.

### 5.8 Instruction Validation Gate 강제 절차
모든 구현 및 설계 지시는 수행 전에 반드시 Instruction Validation Gate를 통과해야 한다.

1. 아래 항목 중 하나라도 해당되면 즉시 실행해서는 안 된다.
2. Invariant와 충돌 가능성이 있는 경우
3. 기존 정책과 불일치하는 경우
4. 요청이 모호하거나 해석이 2개 이상 가능한 경우
5. 기존 코드 구조 원칙(service/engine 분리 등)을 훼손하는 경우
6. 권한, 정책, 계산 로직, 구조, DB에 영향을 주는 경우

### 5.9 사전 검증 시 사용자 확인 의무
Instruction Validation Gate에서 위반 가능성이 확인되면 사용자 확인 없이 임의 해석으로 진행해서는 안 된다. 반드시 아래 형식으로 충돌 사항을 정리하여 사용자에게 재확인을 요청한다.

```md
[Instruction Issue]
- 문제 요약

[충돌 규칙]
- 어떤 invariant 또는 정책과 충돌하는지

[확인 필요]
- 선택지 A
- 선택지 B
```

### 5.10 실행 금지 및 진행 조건
Instruction Validation Gate에서 보류 판정이 난 요청은 사용자 확인 전까지 구현, 설계 변경, 리팩터링, 정책 반영을 진행할 수 없다. 확인이 끝난 뒤에만 구현을 진행한다.

### 5.11 Invariant Check 강제 절차
모든 구조 변경, 로직 변경, 리팩터링 작업은 완료 후 반드시 Invariant Check를 수행한다.

1. Invariant Check 없이 `완료`, `적용 가능`, `merge 가능` 판정을 해서는 안 된다.
2. Invariant 위반이 발견되면 구현 완성도와 무관하게 위반 수정이 우선이다.
3. Invariant 위반에 대한 우회 승인 또는 묵시적 승인 처리는 금지한다.
4. 세션이 바뀌어도 동일 규정을 유지한다.
5. Cursor가 만든 결과물도 동일 기준으로 검사한다.
6. Codex가 만든 결과물도 동일 기준으로 검사한다.
7. ChatGPT 검토 결과도 동일 기준으로 적용한다.

### 5.12 Invariant Check 적용 대상
다음 변경은 모두 Invariant Check 의무 대상이다.

1. 구조 변경
2. 정책 변경
3. 계산 로직 변경
4. 권한 변경
5. 리팩터링

### 5.13 작업 흐름 고정 원칙
구현 작업의 표준 흐름은 다음 순서를 따른다.

1. 설계
2. Instruction Validation Gate
3. 구현
4. Invariant Check
5. 통과 시에만 반영

`필요하면 검사`가 아니라 `반드시 마지막에 검사`를 원칙으로 한다.

### 5.14 검사 누락 시 판정
Invariant Check가 누락된 작업은 완료된 작업으로 간주하지 않는다. 검사 누락 상태에서는 반영 완료, 적용 가능, 병합 가능 판정을 할 수 없다.

#### 5.X.1 목적
본 규칙은 다음을 달성하기 위해 정의한다.

### 5.X+1 리소스 부족 시 대체 실행 규칙 (Fallback Execution Policy)

#### 5.X+1.1 목적
본 규칙은 Cursor 플랜 또는 모델 사용량 제한으로 인해 정상적인 작업 수행이 불가능한 상황에서 작업 중단 없이 대체 실행 경로를 확보하기 위해 정의한다.

#### 5.X+1.2 트리거 조건
다음 조건 중 하나라도 해당되면 본 규칙을 발동한다.

1. 모델 사용량 초과 (Rate limit / quota 초과)
2. 응답 지연 또는 반복 실패
3. 특정 모델 선택 불가 상태
4. Cursor 내부 실행 불안정 상태

#### 5.X+1.3 대체 실행 원칙
위 조건 발생 시 다음 우선순위를 따른다.

```text
1순위: Cursor 내 다른 모델로 전환
2순위: Cursor AUTO 재시도
3순위: VS Code 내 Codex 사용
```

#### 5.X+1.4 VS Code Codex 사용 규칙
Cursor 환경에서 정상 작업이 불가능한 경우, VS Code 내 Codex를 공식 대체 실행 환경으로 사용한다.

#### 5.X+1.5 적용 방식
VS Code Codex 사용 시 아래를 준수한다.

1. 동일 프롬프트 그대로 사용
2. 동일 작업 범위 유지
3. 동일 규정 적용

즉, 실행 환경만 변경될 뿐 규칙은 동일하게 유지한다.

- Instruction Validation Gate 적용
- Invariant Check 적용
- 보고 형식 유지

#### 5.X+1.6 금지 사항
다음은 금지한다.

1. Cursor 실패 후 무작정 다른 모델 시도 반복
2. 규정 없이 VS Code Codex 단독 사용
3. 검증 절차 생략
4. 결과 비교 없이 혼합 반영

#### 5.X+1.7 결과 검증 규칙
VS Code Codex 결과는 반드시 아래를 수행한다.

1. Cursor 기준 규정으로 재검증
2. Invariant Check 수행
3. 기존 코드와 충돌 여부 확인

#### 5.X+1.8 복귀 원칙
Cursor 환경이 정상화되면 즉시 Cursor로 복귀하여 작업을 이어간다.

#### 5.X+1.9 책임 원칙
대체 실행 결과도 동일하게 아래 책임을 유지한다.

- 품질 책임 유지
- 검증 책임 유지
- 보고 의무 유지

## 6. 코드 수정 및 파일 생성 규칙

### 6.1 신규 파일 생성 기준
신규 파일은 다음 경우에만 생성한다.

1. 기존 구조로는 목적 달성이 곤란한 경우
2. 책임 분리가 명확히 필요한 경우
3. 문서화가 기준상 필요한 경우
4. 기존 파일에 무리하게 내용을 누적할 경우

### 6.2 기존 파일 수정 기준
기존 기능과 직접 관련된 수정은 기존 파일 우선 원칙을 따른다. 단, 한 파일에 여러 책임이 과도하게 누적되는 경우에만 분리를 검토한다.

### 6.3 경로 이동 및 리네이밍 제한
파일 또는 디렉터리 경로 이동, 리네이밍, 대량 재배치는 사용자 요청 또는 명확한 승인 없이 수행하지 않는다.

### 6.4 불필요한 리팩터링 금지
버그 수정, 설정 수정, 기능 보완 작업 중 직접 관련 없는 코드 스타일 변경, 네이밍 통일, 포맷 재정리, 대체 라이브러리 전환은 하지 않는다.

### 6.5 계층 분리 원칙
백엔드와 프론트엔드는 각자의 책임 계층을 유지한다.

1. 서비스 로직은 서비스 계층에 둔다.
2. 데이터 접근 로직은 리포지토리 또는 데이터 계층에 둔다.
3. 라우트 또는 컨트롤러는 요청/응답 처리에 집중한다.
4. 계층 침범이 필요한 경우 사유와 영향 범위를 보고한다.

### 6.6 설정 파일 변경 시 의무
환경변수, 포트, CORS, 인증, 빌드 설정, 프록시, 패키지 설정 등 설정 파일을 변경한 경우 반드시 영향 범위를 보고해야 한다.

### 6.7 DB 구조 변경 시 의무
스키마, 마이그레이션, 시드, 제약조건, 컬럼 변경 등 DB 구조에 영향을 주는 작업은 별도 위험 항목으로 보고해야 하며, 사용자 승인 없이 광범위한 스키마 변경을 진행하지 않는다.

## 7. 오류 대응 규칙

### 7.1 추정 결론 금지
오류가 발생했을 때 원인을 추정만으로 단정하지 않는다. 실제 재현과 확인 절차를 거친 후 결론을 제시한다.

### 7.2 기본 점검 순서
오류 대응 시에는 다음 순서를 기본으로 한다.

1. 재현 여부 확인
2. 로그 확인
3. 경로 확인
4. 설정 확인
5. 요청 형식 확인
6. DB 상태 확인

### 7.3 유형별 점검 기준
오류는 다음 유형으로 구분하여 점검한다.

1. `CORS 문제`
허용 origin, credentials, 메서드, 헤더, 프론트 baseURL, 브라우저 origin, 서버 포트를 확인한다.

2. `포트 문제`
프론트와 백엔드 실행 포트, 프록시, 환경변수, README 기준 포트를 확인한다.

3. `인증 문제`
토큰 저장 위치, 요청 헤더, 쿠키 설정, 로그인 엔드포인트, 권한 체크 로직을 확인한다.

4. `DB 문제`
DB 연결 문자열, 마이그레이션 상태, 시드 데이터, 테이블 생성 여부, 트랜잭션 오류를 확인한다.

5. `파일경로 문제`
실제 워크스페이스, 상대경로, 업로드 경로, 정적 파일 경로, import 경로를 확인한다.

### 7.4 미해결 시 보고 기준
오류를 해결하지 못한 경우에는 반드시 아래를 구분하여 보고한다.

1. 무엇을 확인했는지
2. 어디까지 재현했는지
3. 어떤 증거를 확보했는지
4. 무엇이 부족해서 결론을 내릴 수 없는지
5. 다음 확인이 필요한 항목이 무엇인지

### 8.2 Instruction Validation Gate (사전 검증 레이어)

작업 프롬프트(사용자 지시)가 코드/정책/Invariant와 충돌할 가능성이 있는 경우, “구현 전에 차단”하는 사전 검증 단계다. 이 단계는 Invariant Check보다 앞선다.

#### 8.2.1 기본 원칙

1. 모든 구현/설계 지시 수행 전 반드시 검증한다.
2. 아래 항목 중 하나라도 해당되면 즉시 실행 금지:
   - Invariant와 충돌 가능성 있음
   - 기존 정책과 불일치
   - 모호하거나 해석이 2개 이상 가능
   - 기존 코드 구조(예: service/engine 분리 등)를 깨뜨림
3. 위반 가능 시 반드시 아래 형식으로 되묻는다.
4. 사용자 확인 없이 임의 해석으로 진행 금지
5. 확인 후에만 구현 진행

#### 8.2.2 이슈 보고 형식(고정)

```text
[Instruction Issue]
- 문제 요약

[충돌 규칙]
- 어떤 invariant 또는 정책과 충돌하는지

[확인 필요]
- 선택지 A / 선택지 B
```

#### 8.2.3 작업 흐름(고정)

```text
사용자 지시
→ Instruction Validation Gate
→ 설계/구현
→ Invariant Check
→ 통과 시 반영
```

### 8.2 외부 조정자(ChatGPT) 인계 규칙

1. ChatGPT는 코드 수정자가 아니라 **외부 조정자**로 간주한다.
2. Cursor/Codex 등 내부 AI 작업자는 세션을 종료하거나 다른 AI로 전환하기 전에, ChatGPT가 현재 상태를 이해할 수 있도록 아래 최소 정보를 전달하는 것을 원칙으로 한다.
   - 기준 프로젝트 경로
   - 현재 작업 목표
   - 방금 수행한 작업
   - 현재 문제
   - 다음 작업 계획
3. 인계 정보의 **형식은 자유**이나, 위 항목들이 누락되지 않도록 한다. 템플릿 사용을 권장하되 강제하지 않는다.
4. 인계 정보는 가능하면 `docs/BESMA_SESSION_STATE.md`를 기반으로 요약하며, 필요한 경우 해당 문서의 일부를 그대로 인용해도 된다.
5. 다른 AI로 전환되는 경우에도 동일한 원칙을 따르며, ChatGPT가 추후 세션을 복구할 수 있도록 인계 내용을 명시적으로 남긴다.

## 9. 문서화 규칙

### 9.1 docs 폴더 기록 원칙
주요 작업 기준, 운영 기준, 설계 기준, 오류 분석 기준, 보고 기준은 `docs` 폴더에 문서로 남긴다.

### 9.2 README 갱신 원칙
실행 방법, 포트, 설치 절차, 환경변수, 주요 엔드포인트, 개발 절차가 변경되면 `README`를 함께 갱신한다.

### 9.3 프롬프트 자산화 원칙
작업 프롬프트, 운영 프롬프트, 템플릿 프롬프트도 프로젝트 자산으로 관리하며 필요 시 문서화한다.

### 9.4 문서 체계 원칙

1. 통합규정
2. 세부지침
3. 작업프롬프트

## 10. 세부지침 분리 원칙

### 10.1 분리 대상
향후 다음 문서는 별도 세부지침으로 분리 가능하도록 관리한다.

1. 세션 시작 지침
2. 문서 엔진 설계 지침
3. UI/UX 지침
4. 배포 지침
5. 서버 작업 지침
6. DB 마이그레이션 지침
7. 오류 분석 지침
8. 보고서 작성 지침

### 10.2 분리 기준
다음 조건 중 하나 이상에 해당하면 세부지침으로 분리한다.

1. 특정 주제의 절차가 반복적으로 사용되는 경우
2. 오류 또는 혼선이 반복되는 경우
3. 승인 기준 또는 위험 기준이 별도로 필요한 경우
4. 작업자 간 해석 차이가 빈번한 경우

### 10.3 관계 원칙
세부지침은 본 통합운영규정의 하위 문서이며, 상위 규정과 충돌하지 않아야 한다.

## 11. 부칙

### 11.1 개정 원칙
본 규정은 실제 운영 중 발생한 혼선, 오류, 반복 리스크, 협업 비효율을 반영하여 개정할 수 있다. 개정 시에는 개정 사유와 적용 범위를 함께 기록한다.

### 11.2 예외 적용 방식
긴급 대응, 복구 작업, 외부 제약 등으로 본 규정을 예외 적용해야 할 경우, 예외 사유와 범위를 작업 보고에 명시한다. 예외는 해당 세션 또는 해당 작업 범위에 한정한다.

### 11.3 우선순위 관계
문서 간 우선순위는 다음과 같다.

1. 본 통합운영규정
2. 승인된 세부지침
3. 개별 작업프롬프트

개별 작업프롬프트가 본 규정 또는 세부지침과 충돌할 경우, 특별한 승인이나 명시가 없는 한 상위 문서를 우선 적용한다.

## 12. 최종 결정권 규칙

- AI는 작업 분류, 추천 AI, 위임 제안을 수행할 수 있다.
- 그러나 최종 작업 수행 주체는 항상 사용자가 결정한다.
- AI의 추천은 참고사항이며, 강제되지 않는다.
- 사용자가 특정 AI에게 직접 작업을 지시한 경우,
  해당 AI는 추천과 관계없이 작업을 수행해야 한다.
