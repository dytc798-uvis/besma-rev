# BESMA 아키텍처 불변 조건 (Architecture Invariants)

이 문서의 규칙은 **원칙적으로 위반하지 않는다**.  
예외가 필요하면 **오너 승인 + `DECISION_LOG.md`에 새 결정**이 있어야 한다.

---

## Cursor / 협업 도구 운영 규칙 (필수)

1. 작업 시작 전 **`DECISION_LOG.md`와 본 문서(`ARCHITECTURE_INVARIANTS.md`)를 읽는다.**
2. 본 문서 규칙과 **충돌하는 코드·라우트·API 변경을 제안하지 않는다.**
3. 불가피한 예외는 **사용자 확인 후** `DECISION_LOG.md`에 기록한다.
4. 선택이 필요하면 **`OPEN_DECISIONS.md`를 갱신**하고 사용자에게 질문한다.

---

## 1. UI 타입 (고정)

**Rule 1.** 제품의 로그인 후 화면 축은 다음 세 가지뿐이다: **`HQ_SAFE`**, **`SITE`**, **`HQ_OTHER`**.

**Rule 2.** 네 번째 UI 타입이나, 위 세 가지와 **평행한 전용 셸**(별도 글로벌 사이드바 체계)을 새로 만들지 않는다.

**Rule 3.** 라우터에서 `meta.uiType`으로 분기하는 패턴을 깨는 전면 리라이트를 하지 않는다.

---

## 2. 금지 사항 (구조·계약)

**Rule 4.** `safety_platform`, `request_review_action` 등 **문서·TBM·현장 흐름과 별개의 신규 최상위 백엔드 도메인**을 추가하지 않는다. (DECISION-002)

**Rule 5.** 클라이언트가 사용하는 **문서 관련 REST 경로의 “첫 세그먼트”를 바꾸지 않는다** — 즉 **`/documents`**, **`/document-submissions`** 계열을 다른 이름으로 대체하지 않는다. (경로 끝에 파라미터를 붙이는 것은 기존 설계 범위 내에서만.)

**Rule 6.** **`GET /documents/hq-dashboard`**, **`GET /documents/requirements/status`** 등 기존 응답의 **필드 이름·의미를 깨는 변경**을 하지 않는다. (필드 추가는 본 문서만으로 허용되지 않으며, 별도 제품 결정이 필요하다.)

**Rule 7.** 다음 라우트 **경로 문자열을 바꾸지 않는다**: **`/hq-safe/documents`**, **`/site/documents`**. (이름 변경·부모 변경 금지.)

**Rule 8.** “구조 정비” 명목으로 **대량 import 경로 변경·전역 폴더 이동**을 하지 않는다. 필요 시 작은 단위·회귀 테스트와 함께 **별도 결정**이 있어야 한다.

---

## 3. 요구사항 행 상태값 (API 진실)

백엔드 `get_site_requirement_status`가 내려주는 `status` 문자열은 아래 집합에 속한다. **프론트에서 새 코드값을 만들어 서버와 동기화되지 않는 상태를 쓰지 않는다.**

**Rule 9.** 허용되는 `status` 값은 다음과 같다:

- `NOT_REQUIRED` — 해당 기간·주기에 대해 의무 아님  
- `NOT_SUBMITTED` — 제출 없음  
- `SUBMITTED` — 제출됨  
- `IN_REVIEW` — 검토 중  
- `APPROVED` — 승인  
- `REJECTED` — 반려 (재업로드 필요로 해석할 수 있음; **별도 `REWORK_REQUIRED` 코드는 API에 없음**)

**Rule 10.** 화면 한글 라벨은 통일할 수 있으나, **비즈니스 판단·집계는 위 문자열을 기준**으로 한다.

---

## 4. 데이터 한계 (착시 방지)

**Rule 11.** 요구사항 행에 **`due_rule_text`(규칙 설명)**는 올 수 있으나, **항목별 “기한 일시” 필드는 응답 스키마에 없다.** 따라서 **“기한 초과 건수”를 서버 없이 정확히 계산한다고 주장하지 않는다.** (표시는 문구·경고 배지 수준으로만.)

**Rule 12.** `REJECTED` 하나로 **“반려”와 “재조치 필요”를 API가 분리하지 않는다.** UI에서 둘로 나누어 보이게 할지는 **`OPEN_DECISIONS.md` 및 오너 결정**에 따른다.

---

## 5. 제품 구조 원칙 (역할 분담)

**Rule 13.** **문서 취합(요구사항 대비 제출·검토·반려)** 은 운영 우선순위에서 **허브**로 둔다. 결재·TBM·현장·사용자 화면은 **같은 데이터 계열을 보조하거나 심화**하는 역할로 둔다. (신규 API 없이 표현·내비게이션으로 반영.)

**Rule 14.** **`HQ_SAFE`** 는 취합·결재·TBM 모니터·의견·현장·사용자 등 **운영/관리** 중심이다.

**Rule 15.** **`SITE`** 는 제출·모바일 운영·현장 문서 등 **실행** 중심이다.

**Rule 16.** **`HQ_OTHER`** 는 문서·의견 **조회** 중심이다; 기능 추가로 관리자와 동일 권한을 주지 않는다(역할 모델은 기존 `Role` 유지).

---

## 6. UI 구현 스타일 (과도한 확장 금지)

**Rule 17.** 새 화면마다 **새 디자인 시스템을 만들지 않는다.** 이미 있는 `frontend/src/components/product/*`, `frontend/src/styles/hq-stitch.css`, 기존 페이지 패턴을 **우선 재사용**한다.

**Rule 18.** “운영 화면 개선”은 **동일 라우트·동일 API** 안에서 레이아웃·정렬·카피·배지로 수행한다.
